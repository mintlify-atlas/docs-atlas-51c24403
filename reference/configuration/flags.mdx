This page provides a comprehensive reference for configuration flags used across YugabyteDB components. Flags allow you to customize behavior, tune performance, configure security, and manage resources.

## Overview

YugabyteDB uses configuration flags (also called gflags) to control various aspects of database operation. Flags can be set for different components:

- **yb-master** - Master server flags for cluster coordination
- **yb-tserver** - Tablet server flags for data storage and processing
- **yugabyted** - Simplified deployment utility flags

## Flag Types

YugabyteDB flags are categorized by their runtime behavior:

| Flag Type | Description | Restart Required |
|-----------|-------------|------------------|
| **RUNTIME** | Can be changed at runtime using `yb-admin` | No |
| **NON_RUNTIME** | Requires server restart to take effect | Yes |
| **AUTO** | Auto-promoted flags for rolling upgrades | Varies |
| **UNKNOWN** | Stability not yet determined | Potentially |
| **TEST** | Used for testing only, not for production | N/A |

## Setting Flags

### Command Line

Set flags directly when starting a server:

```bash
./bin/yb-master --replication_factor=3 --fs_data_dirs=/data
```

### Configuration File

Use a flagfile to store all flags:

```bash
# master.conf
--replication_factor=3
--fs_data_dirs=/data/master
--master_addresses=10.0.0.1:7100,10.0.0.2:7100,10.0.0.3:7100
```

Start with the flagfile:

```bash
./bin/yb-master --flagfile=/etc/yugabyte/master.conf
```

### Runtime Modification

For RUNTIME flags, use yb-admin:

```bash
# Set a runtime flag on yb-tserver
yb-admin -master_addresses <master-addresses> \
  set_flag <tserver-host>:9100 <flag-name> <new-value>

# Example
yb-admin -master_addresses 10.0.0.1:7100 \
  set_flag 10.0.0.1:9100 rocksdb_compact_flush_rate_limit_bytes_per_sec 268435456
```

### YugabyteDB Anywhere

If using YugabyteDB Anywhere, set flags through the **Edit Flags** feature in the UI.

## Common Flag Categories

### General Server Configuration

<AccordionGroup>
  <Accordion title="Core Server Flags">
    <ParamField path="--version" type="flag">
      Display version and build information, then exit.
    </ParamField>
    
    <ParamField path="--flagfile" type="string">
      Path to configuration file containing flags.
      
      **Example:** `--flagfile=/etc/yugabyte/master.conf`
    </ParamField>
    
    <ParamField path="--max_clock_skew_usec" type="int64">
      Maximum expected clock skew between nodes in microseconds.
      
      **Default:** `500000` (500ms)
      
      **Component:** yb-master, yb-tserver
    </ParamField>
    
    <ParamField path="--time_source" type="string">
      Time source for the database. Set to `clockbound` for highly accurate time.
      
      **Default:** `""`
      
      **Valid values:** `""` (system), `clockbound`
      
      **Component:** yb-master, yb-tserver
    </ParamField>
  </Accordion>

  <Accordion title="Networking Flags">
    <ParamField path="--rpc_bind_addresses" type="string">
      Comma-separated list of network addresses to bind for RPC.
      
      **Default:** Private IP of host
      
      **Component:** yb-master, yb-tserver
    </ParamField>
    
    <ParamField path="--server_broadcast_addresses" type="string">
      Public IP or DNS hostname for server-to-server communication.
      
      **Default:** `""`
      
      **Component:** yb-master, yb-tserver
    </ParamField>
    
    <ParamField path="--dns_cache_expiration_ms" type="int32">
      Duration in milliseconds until cached DNS resolution expires.
      
      **Default:** `60000` (1 minute)
      
      **Component:** yb-master, yb-tserver
    </ParamField>
    
    <ParamField path="--use_private_ip" type="string">
      Policy for using private IPs: `never`, `zone`, `region`.
      
      **Default:** `never`
      
      **Component:** yb-master, yb-tserver
    </ParamField>
  </Accordion>

  <Accordion title="Storage Flags">
    <ParamField path="--fs_data_dirs" type="string">
      Comma-separated list of directories for data storage.
      
      **Required:** Yes
      
      **Component:** yb-master, yb-tserver
      
      **Example:** `/data/disk1,/data/disk2,/data/disk3`
    </ParamField>
    
    <ParamField path="--fs_wal_dirs" type="string">
      Comma-separated list of directories for write-ahead logs.
      
      **Default:** Same as `--fs_data_dirs`
      
      **Component:** yb-master, yb-tserver
    </ParamField>
    
    <ParamField path="--rocksdb_block_cache_size_bytes" type="int64">
      Size of RocksDB block cache in bytes. -1 for automatic sizing.
      
      **Default:** `-1`
      
      **Component:** yb-tserver
    </ParamField>
    
    <ParamField path="--rocksdb_compact_flush_rate_limit_bytes_per_sec" type="int64">
      Rate limit for compaction and flush in bytes per second.
      
      **Default:** `256000000` (256 MB/s)
      
      **Component:** yb-tserver
    </ParamField>
    
    <ParamField path="--db_block_size_bytes" type="int32">
      Size of RocksDB data blocks in bytes.
      
      **Default:** `32768` (32 KB)
      
      **Component:** yb-tserver
    </ParamField>
  </Accordion>

  <Accordion title="Memory Flags">
    <ParamField path="--memory_limit_hard_bytes" type="int64">
      Maximum memory limit in bytes. 0 uses percentage of system RAM.
      
      **Default:** `0`
      
      **Component:** yb-master, yb-tserver
    </ParamField>
    
    <ParamField path="--default_memory_limit_to_ram_ratio" type="double">
      Percentage of RAM to use when memory_limit_hard_bytes is 0.
      
      **Default:** `0.10` (10%)
      
      **Component:** yb-master, yb-tserver
    </ParamField>
    
    <ParamField path="--use_memory_defaults_optimized_for_ysql" type="boolean">
      Optimize memory defaults for YSQL workloads.
      
      **Default:** `false`
      
      **Component:** yb-master, yb-tserver
    </ParamField>
  </Accordion>

  <Accordion title="Logging Flags">
    <ParamField path="--log_dir" type="string">
      Directory for log files.
      
      **Default:** Same as `--fs_data_dirs`
      
      **Component:** yb-master, yb-tserver
    </ParamField>
    
    <ParamField path="--max_log_size" type="int32">
      Maximum log file size in megabytes.
      
      **Default:** `1800` (1.8 GB)
      
      **Component:** yb-master, yb-tserver
    </ParamField>
    
    <ParamField path="--minloglevel" type="int32">
      Minimum log level: 0 (INFO), 1 (WARN), 2 (ERROR), 3 (FATAL).
      
      **Default:** `0`
      
      **Component:** yb-master, yb-tserver
    </ParamField>
    
    <ParamField path="--logtostderr" type="boolean">
      Write logs to stderr instead of files.
      
      **Default:** `false`
      
      **Component:** yb-master, yb-tserver
    </ParamField>
  </Accordion>

  <Accordion title="Cluster Configuration">
    <ParamField path="--master_addresses" type="string">
      Comma-separated list of master RPC addresses.
      
      **Required:** Yes (yb-master)
      
      **Component:** yb-master
      
      **Example:** `10.0.0.1:7100,10.0.0.2:7100,10.0.0.3:7100`
    </ParamField>
    
    <ParamField path="--tserver_master_addrs" type="string">
      Comma-separated list of master addresses for tablet server.
      
      **Required:** Yes (yb-tserver)
      
      **Component:** yb-tserver
    </ParamField>
    
    <ParamField path="--replication_factor" type="int32">
      Number of replicas for each tablet.
      
      **Default:** `3`
      
      **Valid range:** 1-7 (odd numbers recommended)
      
      **Component:** yb-master
    </ParamField>
  </Accordion>
</AccordionGroup>

## Flag Naming Conventions

YugabyteDB flags follow these naming patterns:

- **Component prefix**: `master_*`, `tserver_*`, `ysql_*`, `cql_*`
- **Feature area**: `*_rpc_*`, `*_log_*`, `*_cache_*`, `*_compaction_*`
- **Units**: `*_ms` (milliseconds), `*_bytes`, `*_mb`, `*_usec` (microseconds)

## Best Practices

### Production Deployments

1. **Use flagfiles** for configuration management
2. **Document all custom flags** and their purpose
3. **Test flag changes** in staging before production
4. **Monitor impact** after flag changes
5. **Version control** your flagfiles

### Performance Tuning

1. **Start with defaults** and measure baseline performance
2. **Change one flag at a time** to isolate impact
3. **Monitor metrics** before and after changes
4. **Document performance improvements** for future reference
